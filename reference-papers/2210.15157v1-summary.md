# Conversing with Copilot: Exploring Prompt Engineering for Solving CS1 Problems Using Natural Language
## Summary

This paper studies how well **GitHub Copilot** can solve **introductory programming (CS1)** problems, and whether students can improve its results by **changing only the natural-language prompt**, not the code. The core claim is that prompt engineering is not just a workaround. It may become a real programming skill, especially for novices using AI tools. 

## What the authors did

They tested Copilot on **166 public Python problems** from the **CodeCheck** problem bank. The workflow was simple:

1. Paste the original problem description into VS Code with Copilot enabled.
2. Accept Copilot’s first suggestion.
3. Run it against the autograder.
4. If it failed, **rewrite the problem description with extra natural-language hints** and try again.
5. They did **not** manually edit the generated code during this process. 

The problems covered branches, strings, lists, and 2D arrays. 

## Main results

Copilot solved **79 of 166 problems** on the first attempt, which is **47.6%**. After prompt engineering, it solved **53 more**, leaving **34 unsolved**. That means prompt engineering solved **60.9% of the initially failed problems**. 

By category:

* **Branches:** 4 solved initially, 13 more after prompt edits, 5 still unsolved
* **Strings:** 13 initially, 7 more after edits, 9 unsolved
* **Lists:** 38 initially, 19 more after edits, 8 unsolved
* **2D arrays:** 24 initially, 14 more after edits, 12 unsolved 

## What kind of prompt engineering worked

The most effective prompt changes looked like **short pseudocode in plain English**. Instead of just restating the question, the authors added hints like:

* count matching items
* sum their lengths
* divide total by count

That often pushed Copilot from a wrong solution to a correct one. The authors argue this has pedagogical value because students must translate a vague problem into concrete computational steps.

## Why Copilot still failed

The authors grouped failures into four buckets:

* **Conceptual** problems, where key ideas like swapping neighbors or handling array centers were not well captured
* **Poor prompts**, where the prompt left out a crucial condition
* **Verbose prompts**, where the explanation became too long and hard for the model to use well
* **Ambiguous prompts**, where the task description allowed multiple interpretations 

Their main point is that there is a tension here: you want to make the prompt clearer, but if you make it too detailed, model performance can get worse. 

## Authors’ conclusion

The paper argues that AI code tools will change programming education. Students may spend less time writing code from scratch and more time:

* diagnosing AI failures
* expressing algorithms in natural language
* learning how to prompt effectively

The authors see this as a potentially useful learning activity, not just cheating or shortcut behavior. 

## Limits of the paper

The paper is useful, but narrow:

* only **Python**
* mostly **procedural CS1 problems**
* only one tool setup
* prompt engineering process was somewhat exploratory, not tightly standardized
* Copilot/Codex is **non-deterministic**, so results may vary across time and runs

## Bottom line

This paper shows two things:

1. **Copilot was already strong enough to solve about half of beginner programming problems directly.**
2. **A lot of the remaining failures could be fixed just by improving the prompt.**

That makes prompt engineering a real part of beginner programming now, at least in AI-assisted settings.
